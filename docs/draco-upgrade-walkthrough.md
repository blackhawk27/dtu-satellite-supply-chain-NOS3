# NOS3 cFS Upgrade Walkthrough: 6.7.99 (pre-Draco) to 7.0.0 (Draco Final)

## Overview

This document records every change required to upgrade the NOS3 cFS stack from the
pre-release Draco development build (version string `6.7.99`, internally `rc4+dev370`)
to the final **Draco v7.0.0** release (January 2025).

It also explains the architecture of the NOS3 build system, why the upgrade was more
involved than expected, and what the correct mental model is for future upgrades.

---

## Why It Was Not a Simple "Drop-In" Upgrade

Before diving into the changes, it is worth addressing the assumption that upgrading
cFS should be as simple as replacing `fsw/` while leaving `cfg/` untouched.

That assumption is **partially correct** but breaks down for three reasons in this repo:

### 1. The cfg/ tables are compiled C code, not plain config

The files in `cfg/nos3_defs/tables/` are **compiled C source files**. They include
headers from the apps they command:

```
cfg/nos3_defs/tables/sc_rts001.c
  → #include "ds_msg.h"
  → #include "lc_msg.h"
  → etc.
```

When the apps changed their public types and constants between versions, the table
files broke at **compile time** - not at runtime, and not with a config-file error,
but with actual C type errors. These tables are part of the configuration, but they
are tightly coupled to app API versions.

### 2. The cfg/ mission config was missing macros that the new code requires

The new cFE 7.0.0 and app releases added new required configuration macros. If
`cfe_mission_cfg.h` does not define them, the build fails. This means `cfg/` must be
extended when upgrading, even though you are not changing the flight behavior.

### 3. The MISSION_GLOBAL_APPLIST and CMakeLists.txt are part of the build contract

Several Draco v7.0.0 app releases changed their `CMakeLists.txt` in ways that broke
NOS3-specific include paths. The fix required restoring NOS3 additions in those files.
Similarly, the `targets.cmake` in `cfg/` had to be updated to handle a PSP module
conflict that only appears in the unit test build target.

### The Two-Level Config Sync Problem

NOS3 has a two-level configuration system:

```
cfg/nos3_defs/          <-- the authoritative source you edit
cfg/build/nos3_defs/    <-- the generated copy used at build time (from make config)
```

**Every change to `cfg/nos3_defs/` must also be applied to `cfg/build/nos3_defs/`.**

The `make config` step regenerates `cfg/build/` from `cfg/nos3_defs/`, but only if
you run it. During iterative build-fix loops, if you only edited `cfg/nos3_defs/` and
not `cfg/build/nos3_defs/`, the build would continue to use the old (broken) version.
This caught us multiple times.

## Architecture: What Lives Where

```
nos3/
  fsw/
    cfe/          Core Flight Executive (embedded, not a submodule)
    osal/         OS Abstraction Layer (embedded)
    psp/          Platform Support Package (embedded, with NOS3 nos-linux additions)
    apps/         Heritage NASA cFS apps (embedded copies of upstream releases)
    build/        CMake build output (generated)

  cfg/
    nos3-mission.xml           Top-level mission selector
    nos3_defs/                 Authoritative mission/platform config (YOU edit this)
    build/nos3_defs/           Build-time copy (regenerated by make config)

  components/
    <name>/fsw/cfs/            Per-component cFS apps (NOS3-specific)
    <name>/sim/                Hardware simulators
```

When you run `make build-fsw`, it runs inside a Docker container
(`ivvitc/nos3-64:20251107`) and uses `cfg/build/nos3_defs/` as the mission
definition directory (`MISSION_DEFS` CMake variable).

## Changes by Category

---

### Category 1: Core Framework (cFE, OSAL, PSP)

These three are embedded directories (not git submodules). They were replaced wholesale
with the v7.0.0 final release tarballs from NASA GitHub, with NOS3-specific patches
re-applied on top.

#### cFE: v7.0.0-rc4+dev370 → v7.0.0 final

**Files replaced:** All of `fsw/cfe/` except NOS3-specific patches.

Key API changes introduced between rc4 and the final release:

| Old (6.x / rc4) | New (7.0.0 final) | Location changed |
|---|---|---|
| `CFE_SB_SUB_ENTRIES_PER_PKT` | `CFE_MISSION_SB_SUB_ENTRIES_PER_PKT` | `fsw/apps/sbn/fsw/src/sbn_subs.c` |
| `CFE_MSG_Message_t.Byte` | Cast to `uint8 *` | `fsw/apps/ci_lab/fsw/src/ci_lab_app.c` |
| `CFE_FS_FILE_CONTENT_ID` (not in public headers) | Added to `cfe_fs.h` include chain | `fsw/cfe/modules/core_api/fsw/inc/cfe_fs.h` |

#### OSAL: v6.x → v7.0.0 final

**Files replaced:** All of `fsw/osal/` except the NOS3 `nos` implementation (`fsw/osal/src/os/nos/`).

The v7.0.0 OSAL added several new portable implementation functions that the NOS
platform's `CMakeLists.txt` was not wired up to compile. This caused linker failures
(`undefined reference`) when building `core-cpu1`.

**File changed:** `fsw/osal/src/os/nos/CMakeLists.txt`

```cmake
# Added to NOS_IMPL_SRCLIST:
../portable/os-impl-posix-gettime-monotonic.c   # OS_GetMonotonicTime_Impl
../portable/os-impl-posix-file-allocate.c       # OS_FileAllocate_Impl
../portable/os-impl-bsd-sockets-dns.c           # OS_SocketAddrToString_Impl / FromString_Impl

# Added new conditional block:
if (OSAL_CONFIG_RWLOCK)
    list(APPEND NOS_IMPL_SRCLIST ../posix/src/os-impl-rwlock.c)
else ()
    list(APPEND NOS_IMPL_SRCLIST ../portable/os-impl-no-rwlock.c)
endif ()
```

**Why:** The POSIX and QNX OSAL targets already had these blocks. The NOS target
(which was added by NOS3, not upstream NASA) had not been kept in sync.

#### PSP: rc4+dev → v7.0.0 final (pc-linux portions)

The NOS3-specific `nos-linux` PSP module (`fsw/psp/fsw/nos-linux/`) was preserved
unchanged. Only the upstream pc-linux, shared, and cmake portions were updated.

### Category 2: Heritage NASA Apps (`fsw/apps/`)

Each of the standard NASA cFS apps has an independent release version. The pre-Draco
NOS3 build used older versions. The following were upgraded to their Draco-compatible
releases.

#### LC (Limit Checker)

**Draco change:** Renamed `LC_WATCH_NOT_USED` to `LC_DATA_WATCH_NOT_USED`.

**File changed:** `cfg/nos3_defs/tables/lc_def_wdt.c` (and `cfg/build/` copy)

```bash
# 155 occurrences replaced:
sed -i 's/LC_WATCH_NOT_USED/LC_DATA_WATCH_NOT_USED/g' lc_def_wdt.c
```

**CMakeLists change:** `fsw/apps/lc/CMakeLists.txt`

The Draco LC release removed `include_directories(${APPLICATION_PLATFORM_INC_LIST})`
from its CMakeLists.txt. This variable is a NOS3 addition that points all component
platform_inc directories into the include path - needed because `lc_def_wdt.c`
includes `generic_eps_msgids.h` and other component headers.

```cmake
# Restored NOS3 addition before add_cfe_app():
include_directories(${APPLICATION_PLATFORM_INC_LIST})
```

#### DS (Data Storage)

**CMakeLists change:** `fsw/apps/ds/CMakeLists.txt`

Same `APPLICATION_PLATFORM_INC_LIST` removal. DS tables include `ds_indices.h` which
chains to `fm_msgids.h` → `fm_topicids.h` (a component header). Without the include
path, table compilation fails.

```cmake
include_directories(${APPLICATION_PLATFORM_INC_LIST})
```

#### SC (Stored Commands)

This was the most involved app upgrade. SC v7.0.0 renamed every command type:

| Old type | New type |
|---|---|
| `SC_NoArgsCmd_t` | `SC_NoopCmd_t` / `SC_ResetCountersCmd_t` |
| `SC_RtsCmd_t` | `SC_StartRtsCmd_t` / `SC_EnableRtsCmd_t` |
| `SC_RtsGrpCmd_t` | `SC_StartRtsGrpCmd_t` / `SC_EnableRtsGrpCmd_t` |
| `.RtsId` (payload field) | `.Payload.RtsNum` |

The NOS3 SC RTS mission tables (`sc_ats1.c`, `sc_rts001.c`, `sc_rts003.c`) in `cfg/`
used these old types. Rather than manually updating every command in the tables, they
were replaced with the SC v7.0.0 reference tables from `fsw/apps/sc/fsw/tables/`.

**Files replaced:**
- `cfg/nos3_defs/tables/sc_ats1.c` (and `cfg/build/` copy)
- `cfg/nos3_defs/tables/sc_rts001.c` (and `cfg/build/` copy)
- `cfg/nos3_defs/tables/sc_rts003.c` (and `cfg/build/` copy)

**CMakeLists change:** `fsw/apps/sc/CMakeLists.txt`

Same `APPLICATION_PLATFORM_INC_LIST` restoration needed (SC RTS tables include DS,
LC, FM headers).

> **Note:** The original NOS3 RTS tables commanded actual hardware components (EPS,
> CSS, FSS, IMU, etc.) to initialize on boot. The replacement tables are minimal
> references. Restoring the mission sequences requires updating them to use the new
> SC v7.0.0 API types.

#### FM (File Manager)

**Draco change:** `FM_FREE_SPACE_TLM_MID` was removed. Free space monitoring is now
`FM_MONITOR_TLM_MID`.

**File changed:** `cfg/nos3_defs/tables/to_config.c` (and `cfg/build/` copy)

```c
// Old:
{ FM_FREE_SPACE_TLM_MID, ... }
// New:
{ FM_MONITOR_TLM_MID, ... }
```

#### SBN (Software Bus Network)

**Draco change:** `CFE_SB_SUB_ENTRIES_PER_PKT` renamed to
`CFE_MISSION_SB_SUB_ENTRIES_PER_PKT`.

**File changed:** `fsw/apps/sbn/fsw/src/sbn_subs.c`

```c
// Old:
if (Ptr->Payload.Entries > CFE_SB_SUB_ENTRIES_PER_PKT)
// New:
if (Ptr->Payload.Entries > CFE_MISSION_SB_SUB_ENTRIES_PER_PKT)
```

#### ci_lab (Command Ingest Lab)

**Draco change:** `CFE_MSG_Message_t` no longer has a `.Byte` member in cFE v7.0.0.

**File changed:** `fsw/apps/ci_lab/fsw/src/ci_lab_app.c`

```c
// Old (line 367):
bytes = CI_LAB_Global.NextIngestBufPtr->Msg.Byte;
// New:
bytes = (uint8 *)&CI_LAB_Global.NextIngestBufPtr->Msg;
```

#### CS (Checksum)

**Issue:** `fsw/apps/cs/config/default_cs_tbldefs.h` includes
`cfe_tbl_extern_typedefs.h` which only defines `CFE_TBL_HandleId_t`. It does not
define `CFE_TBL_Handle_t`, which is defined in `cfe_tbl_api_typedefs.h`. This
caused compilation failure in the unit test build.

**File changed:** `fsw/apps/cs/config/default_cs_tbldefs.h`

```c
#include "cfe_tbl_extern_typedefs.h"
#include "cfe_tbl_api_typedefs.h"   /* CFE_TBL_Handle_t -- added */
```

### Category 3: Configuration Files (`cfg/nos3_defs/`)

Remember: every change here must also be applied to `cfg/build/nos3_defs/`.

#### `cfe_mission_cfg.h`

Three additions were required:

```c
// 1. FM v7.0.0 requires this (file monitor table size):
#define CFE_MISSION_MAX_NUM_FILES 50

// 2. cFE FS header description length (used by FM and FS APIs):
#define CFE_FS_HDR_DESC_MAX_LEN 32

// 3. NOS3's cFE time patch uses SECONDS (plural), but cFE defines SECOND (singular):
#define CFE_MISSION_TIME_EPOCH_SECOND 0
#define CFE_MISSION_TIME_EPOCH_SECONDS CFE_MISSION_TIME_EPOCH_SECOND  /* alias */
#define CFE_MISSION_TIME_EPOCH_MICROS 0
```

The `CFE_MISSION_TIME_EPOCH_SECONDS` alias is needed because NOS3 patches
`cfe/modules/time/fsw/src/cfe_time_api.c` to use the plural form, but the cFE 7.0.0
upstream only defines the singular form.

#### `cpu1_platform_cfg.h`

Missing include at the top of the file:

```c
// Added after the header guard:
#include "cfe_mission_cfg.h"
```

The platform config references `CFE_MISSION_SB_MAX_SB_MSG_SIZE` at line 303 but had
no include for `cfe_mission_cfg.h` where that macro is defined. This worked in the
old version by accident (another header happened to pull it in transitively). The new
cFE 7.0.0 header chain no longer does so.

#### `tables/lc_def_wdt.c`

155 occurrences of `LC_WATCH_NOT_USED` renamed to `LC_DATA_WATCH_NOT_USED` (see LC
app section above).

#### `tables/sc_ats1.c`, `sc_rts001.c`, `sc_rts003.c`

Replaced with SC v7.0.0 reference tables due to wholesale type renames (see SC app
section above).

#### `tables/to_config.c`

`FM_FREE_SPACE_TLM_MID` → `FM_MONITOR_TLM_MID` (see FM app section above).

### Category 4: cFE Internal Fixes

These are fixes to the cFE source itself, needed because the v7.0.0 release has
internal include chain gaps.

#### `cfe_fs.h` - missing include for `CFE_FS_FILE_CONTENT_ID`

`CFE_FS_FILE_CONTENT_ID` is defined in `cfe_fs_interface_cfg.h`. This header is not
reachable through the normal `cfe.h` → `cfe_fs.h` include chain. Any code that uses
this constant (cFE TBL unit tests, `cfe_fs_api.c`, `cfe_tbl_load.c`) fails with
`undeclared identifier`.

**Fix:** Added the include to the public API header so all consumers get it
transitively.

**File changed:** `fsw/cfe/modules/core_api/fsw/inc/cfe_fs.h`

```c
#include "cfe_fs_api_typedefs.h"
#include "cfe_fs_extern_typedefs.h"
#include "cfe_fs_interface_cfg.h"   /* CFE_FS_FILE_CONTENT_ID -- added */
#include "cfe_time_api_typedefs.h"
```

Additionally added direct includes to:
- `fsw/cfe/modules/fs/fsw/src/cfe_fs_api.c`
- `fsw/cfe/modules/tbl/fsw/src/cfe_tbl_load.c`

---

### Category 5: Build System (`targets.cmake`)

#### `iodriver` PSP module vs app list conflict

`iodriver` was listed in `MISSION_GLOBAL_APPLIST` in `targets.cmake`. For the
`amd64-nos3` target (production build), this is correct - the nos-linux PSP does not
include `iodriver` in its `psp_module_list.cmake`, so it must be compiled as a cFS
app.

However, the `amd64-posix` target (unit test build) uses the pc-linux PSP, which
**does** include `iodriver` via `psp_module_list.cmake`. This means CMake tries to
create a target named `iodriver` twice, failing with a CMP0002 duplicate target error.

**File changed:** `cfg/nos3_defs/targets.cmake` (and `cfg/build/` copy)

```cmake
# After the main list(APPEND MISSION_GLOBAL_APPLIST ...) block:
# iodriver: pc-linux PSP (amd64-posix) already includes it via psp_module_list.cmake.
# nos-linux PSP (amd64-nos3) does not, so we must add it only for that target.
if (NOT ENABLE_UNIT_TESTS)
    list(APPEND MISSION_GLOBAL_APPLIST iodriver)
endif()
```

## Build Results

### Production build (`amd64-nos3`)

```
make build-fsw
```

Result: **PASS**. `core-cpu1` binary built, 30 tables installed to
`fsw/build/exe/cpu1/cf/`.

### Unit test build (`amd64-posix`)

```
# Inside Docker:
cd fsw/build && MISSION_DEFS=../../cfg/build/ cmake -DENABLE_UNIT_TESTS=true \
  -DCMAKE_INSTALL_PREFIX=... ../cfe && make -j4 mission-install
```

Result: **PASS**. All test runners built.

### Unit tests

```
cd fsw/build/amd64-posix/default_cpu1 && ctest --output-on-failure -O ctest.log
```

Result: **194/198 pass**.

| Test | Status | Root Cause |
|---|---|---|
| `osal-core-test` | FAIL | WSL2 kernel `/proc/sys/fs/mqueue/msg_max=10` too low |
| `queue-test` | FAIL | Same - OSAL queue implementation requires higher mqueue limit |
| `coverage-cs-cs_table_processing` | FAIL | Pre-existing in CS v7.0.0 upstream - `CFE_TBL_Unregister()` call count expectation wrong |
| `coverage-mm-mm_cmds` | FAIL | Pre-existing in MM v7.0.0 upstream - EventID values shifted by 1 |

The OSAL queue failures can be resolved by raising the kernel limit:
```bash
# On the host (WSL2), add to /etc/sysctl.conf:
fs.mqueue.msg_max = 64
# Then: sudo sysctl -p
```

The CS and MM failures are upstream test defects in the v7.0.0 app releases and are
not regressions from this upgrade.

## Lessons Learned and Recommendations for Future Upgrades

### 1. Tables in cfg/ are compiled code, not config

The files in `cfg/nos3_defs/tables/` are C source. They include app API headers.
When you upgrade apps, the tables often need updating too. This is the main reason
a "just replace fsw/" approach does not work cleanly.

**Recommendation:** When upgrading a cFS app, immediately check whether any table
files in `cfg/nos3_defs/tables/` include headers from that app. If so, audit them
for API changes.

### 2. Always sync cfg/nos3_defs/ with cfg/build/nos3_defs/

Running `make config` regenerates `cfg/build/` from `cfg/nos3_defs/`. But during
iterative build-fix loops, running `make config` each time is slow. The workaround
is to apply fixes to both copies manually. If you run `make config` after all fixes,
the build copy will be regenerated correctly.

**Recommendation:** After finishing all fixes, run `make config` and rebuild from
scratch to confirm that `cfg/nos3_defs/` alone (without manual edits to
`cfg/build/`) produces a clean build.

### 3. The APPLICATION_PLATFORM_INC_LIST pattern is NOS3-specific

Upstream NASA releases of LC, DS, and SC removed `include_directories(${APPLICATION_PLATFORM_INC_LIST})` from their CMakeLists.txt because NASA's standard mission does not need component headers during table compilation. NOS3's mission does, because its tables reference component message IDs.

**Recommendation:** When upgrading any app that uses NOS3 component headers in its
tables, check the app's CMakeLists.txt for this include and restore it if missing.
The apps most likely to need it are: LC, DS, SC, HS, HK, TO.

### 4. The OSAL nos implementation needs to track OSAL upstream

The `fsw/osal/src/os/nos/` directory is a NOS3 addition that does not exist in
NASA's OSAL repo. It relies on portable implementation files from
`fsw/osal/src/os/portable/`. When OSAL adds new portable files (as it did in v7.0.0),
the nos CMakeLists.txt must be updated to include them.

**Recommendation:** After any OSAL upgrade, diff
`fsw/osal/src/os/posix/CMakeLists.txt` against
`fsw/osal/src/os/nos/CMakeLists.txt` and check for missing portable file references.

### 5. cFE internal include chains have gaps

The v7.0.0 release has at least one known include chain gap:
`CFE_FS_FILE_CONTENT_ID` is not reachable through `cfe.h`. This is likely to be
fixed in future cFE releases, but for now the fix is already applied in this repo
via the `cfe_fs.h` change.

### 6. Version strings and where to find them

To verify which cFS version is actually built:

```bash
grep "MAJOR\|MINOR\|REVISION" fsw/cfe/modules/core_api/fsw/inc/cfe_version.h
grep "OS_MAJOR_VERSION\|OS_MINOR_VERSION" fsw/osal/src/os/inc/osapi-version.h
```

Expected after this upgrade:
- cFE: MAJOR=7, MINOR=0, REVISION=0
- OSAL: OS_MAJOR_VERSION=7, OS_MINOR_VERSION=0
