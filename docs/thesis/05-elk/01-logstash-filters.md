# Logstash filter pipeline

The Logstash filter section in `nos3/elk/logstash.conf` (about
1300 lines, sitting between the `input` and `output` blocks)
turns four raw input streams into documents that Kibana can
query. This document walks the pipeline in the order Logstash
applies its rules. Filenames and line numbers are against the
current revision; the line numbers are approximate.

The four input types and where they come from are described in
[00-pipeline.md](00-pipeline.md); this document assumes that
framing and focuses on what each filter rule does.

## Preamble: timestamp normalisation

The first conditional in the filter block normalises the
`timestamp` field on `software_bus` documents. The capture
script `god_view_capture.py` writes timestamps as floats
(`1777207816.124`), which Jackson serialises in scientific
notation for sufficiently large values. The Elasticsearch
`epoch_millis` date parser rejects that form. The filter
coerces the float to an integer-millisecond long that the
index template's `epoch_millis` format accepts cleanly.

This block runs unconditionally for `software_bus` and is a
no-op for the other three types. It exists because the
alternative (changing the capture script's output format) would
break the `hk_decoded` consumers downstream, which expect the
timestamp in seconds.

## Layer 4 tag: `gs_uplink`

Each `gs_uplink` document is tagged `gnss_ping_uplink` and
otherwise left as-is. This is a single-line filter at the top of
the section because every record on this input is by definition
a ping uplink; the tag exists so that downstream dashboard
panels can join uplink documents to the matching `gnss_ping_pong`
events on the bus.

## Layer 1: Software Bus enrichment

The bulk of the filter section dispatches on
`if [type] == "software_bus"`. Three subsections inside it:

### MID translation

Three `translate` filters resolve `msg_id` (hex string) against
the YAML dictionaries bind-mounted at
`/etc/logstash/mid/mid_to_name.yml`,
`/etc/logstash/mid/mid_to_layer.yml`, and
`/etc/logstash/mid/mid_to_subsystem.yml`. The output fields are
`msg_name`, `layer`, and `subsystem` respectively. The
dictionaries themselves are generated by `make config`
(`scripts/mid/gen_mid_registry.py`) or seeded from
`nos3/elk/seed_mid/` when the generator fails (see
[../03-communication/02-fsw-software-bus.md](../03-communication/02-fsw-software-bus.md)
for the load-bearing reason this fallback exists).

The `layer` field uses the taxonomy declared in the top of
`logstash.conf`:

- `FSW_CORE`: cFE OS services (ES, EVS, SB, TBL, TIME).
- `FSW_HERITAGE`: standard cFS apps (TO, CI, DS, FM, CF, LC,
  SC, SCH).
- `COMPONENT`: hardware component cFS apps.
- `RESEARCH`: custom research apps (`BEACON_APP` at the moment).
- `UNKNOWN`: any MID the registry does not resolve.

The `UNKNOWN` value is the canary that the dictionary is stale.
A new component MID that does not appear in the dictionaries
will land in Kibana with `layer = UNKNOWN`, which is the
trigger for the operator to rerun `make config` or fix the
generator.

### Attack-MID tagging

A small set of `mutate { add_tag }` rules (around lines 150 to
170) tag specific MID and function-code combinations:

- `noisy_app_spam_target` on any document whose `msg_id` falls
  in a narrow seven-MID set declared at `logstash.conf:150-155`
  (`0x1806` ES, `0x1801` EVS, `0x1803` SB, `0x1805` TIME,
  `0x1804` TBL, `0x1884` CI, `0x1880` TO). These are the
  highest-impact saturation targets, not the full carpet-bomb
  range. `noisy_app.c` currently iterates every MID in
  `0x0000..0x1FFF`, so most storm packets do not receive this
  tag; the storm is detectable instead through `attack_armed`
  and `sb_pipe_overflow` on the EVS stream.
- `beacon_cmd` on any command document with the
  `MALWARE_TRIGGER_MID` (`0x18F0`).
- `gnss_ping_uplink` mirroring the Layer 4 tag onto the
  matching command-MID documents on the bus.

These tags are the join keys between the four-input view and
the user-facing dashboards; they do not change the underlying
document contents.

## Layer 2: simulator and event logs

The `if [type] == "system_log"` branch is the largest single
block in the file. It applies a base grok to the line format,
then dispatches on the extracted `app` or `subsystem` field
into per-component sub-blocks.

### Base grok

A single grok pattern at line 217 extracts `timestamp`,
`log_level`, and `sim_component` from the line. A subsequent
`mutate` renames the post-message field to `sim_message` so
that the index mapping's keyword sub-field
(`ignore_above: 1024`) applies cleanly.

A second grok (line 240) handles the cFS Event Service variant:
`evs_app_name`, `evs_event_id`, `evs_event_type_num`, and
`evs_message`. A third (line 251) handles the older EVS line
shape that some cFS builds emit. A small ladder of mutates
turns the numeric event type into a human-readable
`evs_severity` value:

- `1 -> DEBUG`
- `2 -> INFORMATION`
- `4 -> ERROR`
- `8 -> CRITICAL`

A final fallback mutates every unmatched EVS line to
`INFORMATION` so the field is always populated and the
dashboard's severity facet is never empty.

### NOISY_APP attack tagging

The block at lines 328 through 347 handles the `noisy_app` event
stream specifically. The `attack_app` field is set to
`NOISY_APP` on any `system_log` document whose `evs_app_name`
matches. Three tags are then added on specific event ids:

- Event id 1 (the startup event "Active. Listening on BEACON
  MID 0x18F0.") tags `attack_loaded`.
- Event id 2 (the threshold event "OMNIDIRECTIONAL STORM
  INITIATED") tags `attack_armed`.
- Event id 3 (each `PING #N intercepted` event) tags
  `attack_trigger_ping`.

A fourth tag, `sb_pipe_overflow`, is set on any `cfe_sb` event
named `SB_PIPE_OVERFLOW`. The four tags together produce a
self-contained Kibana saved search that finds the entire
attack window from arming through carpet-bomb.

### Mission-manager mode extraction

The mgr-specific block (around line 406) extracts
`mgr_mode_num` from the EVS event text, converts to integer, and
applies the two human-readable aliases:

- `mgr_mode_num == 1 -> mgr_mode = SAFE`
- `mgr_mode_num == 3 -> mgr_mode = SCIENCE`

The two reboot-flavoured modes (`2` and `4`) are intentionally
left without a label so they appear in Kibana as raw integers; a
reboot-in-progress is visible as a transient transition through
those values.

### Limit-checker action points

A short block (around line 420) extracts `lc_ap_num` and
`lc_ap_to` from the `LC` app's action-point events. When the
action point transitions to `FAIL`, the document is tagged
`lc_actionpoint_fired`. This is the join key between the LC
table the IDE is currently displaying
(`cfg/build/nos3_defs/tables/lc_def_wdt.c`) and any Kibana
analysis of limit-checker firings.

### Health-and-safety failures

Around line 437 the HS app's failure events are tagged
`hs_app_failure`. This is the line that surfaces
`noisy_app`'s post-arm CPU starvation: the HS app emits an event
when an app misses its expected execution cadence, and the
storm causes exactly that.

### Per-subsystem numeric extraction

The remainder of the `system_log` branch is a series of grok
filters keyed off `subsystem`, each extracting one component's
numeric telemetry into top-level Kibana fields:

- `EPS`: `eps_solar_power_w`, `eps_power_used_w`,
  `eps_battery_wh`, `eps_battery_mv`, `eps_solar_array_enabled`.
- `RW`: `rw_momentum`.
- `GPS`: `gps_lat`, `gps_lon`, `gps_alt`, `gps_ecef_x/y/z`,
  `gps_abs_time`, `gps_valid`, `gps_location` (a `geo_point`
  composed from `gps_lat` and `gps_lon`).
- `CPU_MON`: `TOTAL_CPU_PCT`, `CPU_PCT`, `NUM_PIDS`.

Each of these is a small grok pattern followed by a `mutate`
to convert the captured strings to numeric types. The cost of
adding a new numeric field is the marginal grok plus the
corresponding field in the index template (see
[02-elasticsearch-mappings.md](02-elasticsearch-mappings.md)).

## Layer 3: decoded housekeeping

The `if [type] == "hk_decoded"` branch is short because the
input is already JSON. The block does three things:

- Renames the input `timestamp` to `sim_timestamp_unix` so the
  Layer 1 millisecond normalisation does not stomp on it.
- Computes `sim_timestamp_iso` from `sim_timestamp_unix` for
  date-histogram panels that prefer the ISO form.
- Forwards every other field through as-is, relying on
  Elasticsearch's dynamic mapping to type them.

## Cross-cutting tail: mode-change detection

The final block of the filter section is a Ruby filter
maintaining a per-subsystem state map. For every `MGR` document
that carries `spacecraft_mode`, it compares against the last
seen value for the same subsystem and emits a
`spacecraft_mode_change` tag plus `spacecraft_mode_prev` /
`spacecraft_mode_new` / `mode_change_label` fields the moment
the value transitions. The state map is per-Logstash-process,
which is fine for the single-pipeline, single-instance
deployment in this fork.

The same block then projects the human-readable mode label
(`SAFE`, `SAFE_REBOOT`, `SCIENCE`, `SCIENCE_REBOOT`, `OTHER`)
into `spacecraft_mode_name`, and mirrors that label onto
`GENERIC_GNSS` housekeeping documents based on their
`in_science_mode` flag. The mirroring exists so that the
"Round-trip log" panel on the GNSS-GS-validation dashboard can
show a mode column without joining across documents.

## What the filter section deliberately does not do

The pipeline does not attempt to:

- Cryptographically verify any document. Logstash treats every
  input as trusted; the threat model is documented in
  [../03-communication/06-telemetry-to-elk.md](../03-communication/06-telemetry-to-elk.md).
- Aggregate across documents at index time. All aggregation is
  done in Kibana, against the raw documents.
- Apply a schema beyond the field names. Dynamic mapping in
  Elasticsearch handles the type inference (with the small
  set of explicit overrides in the index template).

These omissions are intentional. The pipeline is the forensic
layer; the analytical layer is Kibana.
