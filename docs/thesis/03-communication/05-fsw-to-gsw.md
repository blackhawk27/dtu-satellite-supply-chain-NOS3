# 03.05 Flight Software to Ground Software (CCSDS over CryptoLib)

This document describes the modelled cryptographic boundary in
the testbed: the operator console talks to the flight software
through CCSDS-framed packets, wrapped by CryptoLib's SDLS
authenticated layer. It pins to figure F6 in
[`../08-figures/figures.md`](../08-figures/figures.md).

## 1. The two endpoints

The `nos3-cosmos` container runs the COSMOS operator console
plus an in-process CryptoLib instance. The `nos3-fsw` container
runs the cFE binary, including two cFS apps that own the
ground link:

- **CI_LAB**: command ingress. Receives CCSDS command packets
  from the ground link and injects them onto the Software Bus
  by MID. MID `0x1884` is the CI_LAB command MID.
- **TO_LAB**: telemetry egress. Subscribes to a configured set
  of telemetry MIDs on the SB and ships the packets back to the
  ground link.

Between them sits the SDLS-wrapped TCP socket. CryptoLib runs
on both ends of that socket: COSMOS-side it wraps outgoing
commands and unwraps incoming telemetry; FSW-side it does the
mirror operation.

The exact uplink and downlink port numbers are generated by the
COSMOS configuration step at `make config` time and live in
`nos3/gsw/cosmos/`. Defaults vary by COSMOS target; the EO1
mission uses ports in the 5010 / 7779 range plus the HTTP UI
on port 2902.

### 1a. CmdTlmServerHeadless (2026-05-03)

In normal upstream NOS3, COSMOS launches via `xvfb-run ruby CmdTlmServer`. That stopped working when the cosmos container's apt snapshot could not install xvfb. The post-thesis fix lives in two files:

- `nos3/gsw/cosmos/tools/CmdTlmServerHeadless` (new, +28 lines, commit `c6c79140`): a pure-Ruby wrapper that runs the CmdTlmServer logic without Qt/Xvfb. No GUI, no display server.
- `nos3/scripts/ci_launch.sh` cosmos branch: switched the entrypoint from `xvfb-run ruby CmdTlmServer` to `ruby CmdTlmServerHeadless`, removed the X11 passthrough for the cosmos container, and added a `-p 5113:5013/udp` port map so the host can reach the listener.

When the GUI is genuinely needed (manual debugging), the `cosmos-gui` make target still launches the X-dependent server; see [`../05-elk/03-kibana-dashboards.md`](../05-elk/03-kibana-dashboards.md).

### 1b. Telemetry Mirror Path (2026-05-03)

COSMOS's DEBUG listener on UDP 5013 sits inside the cosmos container's netns. The default `god_view_capture.py` only sent each downlink packet to the host's god_view receiver, so COSMOS got nothing on its DEBUG listener. The fix lives at `nos3/scripts/god_view_capture.py` (commit `f45ad877`):

- New env var: `NOS3_TLM_MIRROR_DEST` (`host:port` or `ip:port`).
- After every successful god_view write, the script duplicates the bytes to the mirror destination.
- `ci_launch.sh` sets the env var to the COSMOS container's mapped UDP port.
- A separate fix in the same file uses `math.isnan(x)` to scrub NaN values from early-tick 42 packets that would otherwise break Logstash's JSON parser.

Without both halves the GS Hello-Flow loop times out every cycle (`ECHO_TIMEOUT_S = 5.0 s` in `gnss_gs_responder_task.rb`). See [`../04-apps/gnss.md#4a-gs-hello-flow--responder-loop-2026-05-03`](../04-apps/gnss.md#4a-gs-hello-flow--responder-loop-2026-05-03) for the responder side.

## 2. The CCSDS frame

Every packet on the ground link is a CCSDS Space Packet:

- A 6-byte primary header with the 16-bit MID, a sequence
  count, and a packet-data length.
- The user-data field, which for command packets includes a
  command code byte and a packet-secondary-header timestamp,
  and for telemetry packets includes the app's HK or telemetry
  payload structure.

The MID (Message ID) decoded by CI_LAB and the SB router is the
same 16-bit value described in
[`02-fsw-software-bus.md`](02-fsw-software-bus.md). The bridge
between the ground side and the FSW side is therefore a single
field: an operator on the ground knows exactly which app will
receive their command by looking at the high four bits and the
topic ID.

## 3. The SDLS wrapper

CryptoLib implements the CCSDS Space Data Link Security (SDLS)
protocol. SDLS adds:

- A Security Header that identifies the security context and
  carries an initialisation vector for any encryption.
- An optional encryption layer over the user data.
- A trailer carrying a Message Authentication Code (MAC)
  computed across the frame.

The keys are loaded from the CryptoLib configuration at GSW
build time. The keys are deliberately not in the repo;
re-generating them is part of `make gsw`. SDLS is the
attacker-distinguishing primitive at the FSW boundary: an
attacker who can place packets on the wire but does not have
the SDLS keys cannot forge a frame that CI_LAB will accept.

The thesis attack model treats SDLS as a true integrity
boundary. Attacks that breach this layer in the testbed are
attacks that originate inside COSMOS itself (a compromised
COSMOS plug-in or stored command table), not attacks on the
wire.

## 4. The command path

Walk through a `MGR_SET_MODE_CC` command (the MGR app's
mode-change command):

1. The operator clicks the button in the COSMOS UI tab. The
   browser POSTs to the COSMOS HTTP UI on port 2902.
2. COSMOS resolves the command to a CCSDS frame: MID
   `0x19F8 | 0x00 = 0x19F8`, command code `MGR_SET_MODE_CC`,
   command payload (target mode).
3. COSMOS submits the frame to the per-target uplink socket.
   CryptoLib wraps the frame in SDLS and forwards the encrypted
   bytes over the TCP socket to the FSW's CI_LAB.
4. CI_LAB unwraps the SDLS layer (CryptoLib runs in the FSW
   container as a linked library), recovers the raw CCSDS
   frame, and calls `CFE_SB_TransmitMsg` on the SB with the
   recovered frame.
5. The SB looks up subscribers to MID `0x19F8`. The MGR app's
   command pipe is a subscriber; the SB delivers the frame
   into MGR's pipe.
6. The MGR app's main loop pops the message, dispatches by
   command code to its mode-change handler, and updates the
   mission-state HK packet that will be published on the next
   wakeup tick.

## 5. The telemetry path

The downlink path is symmetric:

1. The component app (for example, `generic_eps`) publishes a
   HK packet on its TLM MID on the SB at the wakeup tick.
2. The Housekeeping aggregator (`HK`) subscribes to the
   per-component HK MIDs, packs them into a combined HK packet,
   and publishes that on its own TLM MID.
3. TO_LAB has subscribed to a configured downlink set of MIDs;
   it copies each delivered packet to the per-target downlink
   socket.
4. CryptoLib wraps the outgoing CCSDS frame in SDLS and forwards
   it over the TCP socket to COSMOS.
5. COSMOS unwraps, parses the CCSDS, and updates the operator
   console with the per-field values.

The TO_LAB downlink set is configured at FSW build time in
`nos3/components/<name>/fsw/cfs/tables/` and aggregated in the
TO_LAB configuration. Adding a TLM MID to the downlink visible
to the operator requires editing TO_LAB; this is part of every
new component's deep-dive entry.

## 6. Failure modes

The two failure modes most often hit in this layer:

- **Key mismatch**: a fresh `make gsw` regenerates CryptoLib
  keys; if the FSW container was not rebuilt afterwards, the
  FSW's CryptoLib instance has the old keys, and every command
  fails the SDLS MAC. The diagnostic is in the COSMOS log
  ("MAC verification failed"); the fix is `make` (which
  rebuilds both in lock-step).
- **TO_LAB downlink subscription gap**: a component is added
  to the FSW and emits HK on the SB, but TO_LAB was not
  reconfigured to subscribe. The HK packet circulates on the
  SB but never reaches the ground. The diagnostic is that the
  COSMOS UI shows nothing for the new component and the ELK
  Software-Bus tap (which has its own subscription) still
  sees the packet. The fix is to add the TLM MID to
  TO_LAB's downlink set and rebuild.

## 7. Attack surface at this layer

Three positions are visible in figure F8:

- **COSMOS-internal**: the operator's command table, the
  stored RTS table at
  `nos3/cfg/nos3_defs/tables/sc_rts003.c`, parameter tables
  copied into TBL service. An attacker with COSMOS access can
  issue any command the operator can; the SDLS layer is
  authenticated at the COSMOS boundary, not at the operator's
  console.
- **CryptoLib supply chain**: a tampered CryptoLib binary on
  either end can selectively drop packets, leak keys, or
  forge MACs on attacker-supplied frames. The thesis treats
  this as a build-time supply-chain hit.
- **TO_LAB filter**: a tampered TO_LAB subscription set can
  drop telemetry from the operator's view while the FSW still
  emits it. This is the FSW-side analogue of the Logstash
  filter attack at the ELK boundary
  ([`06-telemetry-to-elk.md`](06-telemetry-to-elk.md)).
